## 1. 아키텍처 스타일 — 모듈러 모놀리식

**개념.** 배포 단위는 하나(모놀리식)지만, 내부를 바운디드 컨텍스트(업무 영역별 자율 구역)로 강하게 나누고 그 경계를 코드로 강제하는 방식. 우리는 identity·project·resource·maintenance·notification·common 6개 도메인 모듈 + 지원 모듈 2개(chat=BFF, mcpconfig=MCP 보안·관측)로 나누고, Spring Modulith가 경계 위반(남의 모듈 내부 직접 참조)을 테스트에서 잡아낸다.

**대안 비교.**

| 방식 | 장점 | 우리 조건에서의 문제 |
|------|------|---------------------|
| **MSA** (서비스별 분리 배포) | 팀별 독립 배포·확장 | 40명 규모에 네트워크·분산 트랜잭션·배포 파이프라인 복잡도만 늘고 얻는 게 없다. 1인 개발엔 독. 기획서도 명시적 제외 |
| **단순 모놀리식** (경계 없음) | 가장 단순한 시작 | 모듈이 서로 자유롭게 참조하다 "진흙공(big ball of mud)"이 된다. 2차 확장 때 어디를 건드려도 전체가 흔들림 |
| **모듈러 모놀리식** ✅ | 모놀리식의 단순함 + 경계의 규율 | — |

**왜 이것인가.** MSA의 이득(독립 배포·팀 분리)은 우리에게 존재하지 않는 문제의 해결책이고, MSA의 비용(운영 복잡도)은 1인 개발에게 치명적이다. 반대로 경계 없는 모놀리식은 몇 년 뒤 유지보수 불능이 된다. 모듈러 모놀리식은 **지금은 하나로 단순하게 돌리고, 나중에 필요해지면 모듈 단위로 떼어낼 수 있는** 중간 지점이다. 2차에 sales·MS사업본부를 얹을 때 추가가 국소적이라는 것(NFR-11)이 실질 이득.

**정확히 알고 쓰기.** "모놀리식 = 나쁜 것, MSA = 좋은 것"이 아니다. 아키텍처는 조직 규모와 문제의 함수이고, 경계는 배포 단위가 아니라 **코드 규율**로도 지킬 수 있다는 것이 이 스타일의 핵심 통찰이다.

## 2. 모듈 간 통신 — 직접 호출과 이벤트

모듈을 나눴으니 모듈끼리 어떻게 대화하는지가 다음 질문이다. 우리는 **두 방식을 목적에 따라 구분**해서 쓴다. 기본은 직접 호출, 이벤트는 사후 fan-out에만.

### 2-1. 직접 호출 (포트 + 의존성 역전) — 기본값

**개념.** 호출하는 모듈이 인터페이스(포트)를 정의하고, 대상 모듈이 그것을 구현한다. 호출 측은 구현체가 아니라 추상에만 의존(DIP)하며, 동기로 결과를 받는다.

**언제.** ① 결과가 즉시 필요한 질의 — 프로젝트 화면이 담당자 이름을 identity에 묻기. ② 반드시 함께 성공/실패해야 하는 명령 — 완료 프로젝트 → 유지보수 이관은 상태 전이와 초기 이력 생성이 **한 트랜잭션**이어야 하므로 project가 maintenance를 동기 호출한다.

### 2-2. 이벤트 — 사후 fan-out에만

**개념.** 발행 모듈은 "무슨 일이 일어났다"만 알리고 구독자의 존재를 모른다. 하나의 사건이 여러 독립 반응으로 퍼질 때 쓴다. 예: 배정 저장 → `MemberAssignedToProject` 발행 → resource(가동률 재계산)와 notification(알림 적재)이 각자 구독.

**신뢰성 장치 3종.** ① `AFTER_COMMIT` — 저장이 확정된 뒤에만 발행(헛방송 방지). ② Event Publication Registry — 미처리 이벤트를 DB 대장에 남겨 재시도(유실 방지, 트랜잭션 아웃박스 패턴). ③ 멱등 재계산 — "그 달을 다시 계산"이라 중복 수신해도 안전.

### 2-3. 왜 카프카(외부 브로커)가 아닌가

Kafka·RabbitMQ는 **프로세스 사이**를 잇는 장비다. 우리는 발행자와 구독자가 같은 앱 안에 있으므로 Spring Modulith의 인프로세스 이벤트로 충분하고, 브로커는 운영할 인프라만 하나 늘린다. 단일 인스턴스 + 동기 처리라 순서 보장도 공짜다. MSA로 쪼개는 날에만 브로커로 외부화한다.

**정확히 알고 쓰기 — 가장 흔한 오용.** "느슨한 결합 = 전부 이벤트"가 아니다. 질의나 원자성이 필요한 명령까지 이벤트로 바꾸면 제어 흐름 추적만 어려워지는 과설계다. 판별 기준 한 줄: **"답이 즉시 필요하거나 반드시 함께 성립해야 하면 동기 직접 호출, 하나의 사건이 여러 사후 반응으로 퍼지면 이벤트."**

## 3. 백엔드 — Spring Boot (Java)

**개념 구분부터.** Spring은 Java 진영의 사실상 표준 애플리케이션 프레임워크(DI, 트랜잭션, 보안 등)이고, Spring Boot는 그 Spring을 "설정 지옥 없이" 쓰게 해주는 포장이다 — 의존성만 넣으면 자동 구성, 웹서버 내장, 실행 가능한 jar 하나로 패키징. 오늘날 "Spring 쓴다" = "Spring Boot 쓴다"와 동의어.

**왜 Java/Spring 계열인가.**

1. **국내 기업 백엔드의 표준** — 인력 풀이 가장 넓다. 사내 도구는 만드는 기간보다 유지하는 기간이 길므로 "누구든 이어받을 수 있는 스택"의 가치가 크다.
2. **도메인 성격과의 궁합** — PMS는 권한 판정·트랜잭션·상태머신·감사 로그 등 **규칙이 많은 업무 시스템**이다. 정적 타입(컴파일 시점 실수 차단) + 성숙한 트랜잭션 관리 + 검증된 보안 프레임워크 조합이 이런 도메인의 사고를 줄인다.
3. **설계 결정과 생태계 부품의 1:1 대응** — 모듈 경계는 Spring Modulith, 영속성은 JPA, 인증은 Spring Security, MCP 서버/클라이언트는 Spring AI. **아키텍처의 핵심 결정마다 같은 가족의 성숙한 부품이 있다.** 특히 Spring Modulith 급의 모듈러 모놀리식 지원은 다른 생태계에 드물다.

**대안 비교.**

| 대안 | 강점 | 우리 조건에서 밀리는 이유 |
|------|------|---------------------------|
| Node.js (NestJS) | 프론트와 언어(TS) 통일, I/O 중심 서비스 | ORM·트랜잭션 성숙도가 JPA에 못 미침. 규칙 많은 업무 시스템보다 실시간·게이트웨이에 강점. 국내 사내 시스템 인력 풀 얇음 |
| Python (Django/FastAPI) | 빠른 프로토타이핑, AI 생태계 | 동적 타입 — 복잡한 권한 규칙이 런타임에야 터짐. 장기 유지보수 규율 부족 |
| Kotlin + Spring | 같은 생태계를 더 간결한 언어로 | 가장 아까운 대안. 설계는 동일하게 유지 가능 — 팀 표준이 Kotlin이면 바꿔도 됨 |
| Go, .NET | 각자 훌륭 | 국내 사내 시스템 맥락에서 인력·레퍼런스·기존 자산 어느 축에서도 우위 근거 없음 |

## 4. 영속성 — JPA

**개념.** JPA(Java Persistence API)는 ORM(객체-관계 매핑) 표준 — 자바 객체와 DB 테이블 사이의 변환을 프레임워크가 대신한다. SQL을 손으로 쓰는 대신 객체를 저장·조회하면 SQL이 생성된다. 구현체는 Hibernate, 사용 편의층은 Spring Data JPA.

**대안 비교.** MyBatis(SQL 매퍼 — SQL을 직접 쓰고 결과만 객체로) 또는 순수 JDBC가 대안이다. MyBatis는 복잡한 통계성 SQL을 정밀 제어할 때 강한데, 우리 도메인은 애그리거트 단위의 CRUD + 소수의 집계라 **객체 모델 중심(DDD)** 인 JPA가 맞다. 특히 애그리거트·값객체·도메인 이벤트 같은 우리 도메인 모델 설계가 JPA의 사용 방식과 자연스럽게 맞물린다.

**정확히 알고 쓰기.** 우리 설계는 "도메인 객체와 JPA 매핑을 분리해 도메인 순수성 유지(NFR-10)"를 요구한다 — 도메인이 JPA를 import하지 않는다. JPA는 도구일 뿐 도메인의 주인이 아니라는 규율이고, 덕분에 도메인 테스트가 DB 없이 순수 객체로 돈다. 또 낙관적 락(`@Version`), 지연 로딩, N+1 문제 같은 JPA의 동작 원리는 "몰라도 돌아가지만 모르면 사고 나는" 영역이니 학습 필수.

## 5. DB — 관계형, 그중 PostgreSQL

### 5-1. 왜 관계형(RDB)인가

우리 데이터는 관계형 모델의 교과서적 적용 대상이다.

1. **관계 중심** — 사람↔팀, 프로젝트↔배정, 배정↔가동률, 유지보수↔원프로젝트. 전부 "무엇이 무엇과 어떤 관계인가"다.
2. **형태 확정적** — 프로젝트에는 반드시 상태·기간·계약M/M이 있다. 스키마가 부담이 아니라 보호 장치.
3. **정합성·원자성 필수** — 중복 배정 금지는 unique 제약으로, 이관의 원자성은 트랜잭션(ACID)으로 DB가 최종 보장한다.
4. **임의 조합 질의 일상** — "이번 달·이 본부·보정 100% 초과·원인 배정과 함께" 같은 예상 못 한 조합을 SQL이 소화한다. 조회 중심 시스템의 본체 기능.

NoSQL(문서형·키값·그래프)은 열등한 게 아니라 **다른 문제**(가변 스키마, 초대규모 수평 확장, 깊은 관계 탐색)를 푸는 도구다. 우리 조건에서 NoSQL이 푸는 문제는 없고, NoSQL이 포기하는 것(조인·트랜잭션·즉석 집계)은 전부 필요하다.

### 5-2. 왜 PostgreSQL인가

| 근거 | 내용 |
|------|------|
| 비용·라이선스 | 완전 오픈소스, 가장 관대한 라이선스. 사내 40명 도구에 Oracle/MSSQL 비용은 정당화 불가 |
| **스키마 논리 분리** | 한 DB 안에 `identity.*`, `project.*` 같은 스키마(이름공간)를 두는 게 일급 개념 — 설계서의 "모듈별 스키마 분리" 결정을 문자 그대로 지원. MySQL은 스키마=데이터베이스라 같은 그림이 안 나옴 |
| 동시성 모델(MVCC) | "읽는 사람이 쓰는 사람을 막지 않는" 방식 — 조회가 대부분이고 소수가 갱신하는 우리 패턴에 이상적. 낙관적 락 전략과 궁합 |
| 트랜잭션 엄격성 | DDL까지 트랜잭션에 포함 — 마이그레이션이 반쯤 적용된 채 남지 않음. Event Registry·append-only AuditLog의 바닥으로 신뢰 가능 |
| JSONB | 가변 구조 데이터(감사 diff 등)를 관계형 안에서 저장·질의·인덱싱 |
| 확장성 보너스 | pgvector로 벡터 검색(RAG 임베딩)까지 같은 DB에서 가능 — Spring AI 공식 지원 |

MySQL/MariaDB도 이 규모에선 돌아가지만, 위 항목마다 조금씩 밀리고 특히 스키마 분리는 구조적으로 다르다. 테스트는 H2·Testcontainers — Testcontainers 덕에 테스트도 진짜 PostgreSQL로 돌려 "테스트와 운영 DB가 달라 생기는 버그"를 막는다.

## 6. 프론트엔드 — React (+ Vite, TypeScript)

**전제 정리.** 이 규모의 관리형 화면에서는 React든 Vue든 기술적으로 다 된다. 성능·기능이 아니라 **유지보수 리스크**로 갈리는 선택이다.

**왜 React인가.**

1. **생태계·레퍼런스 최대** — 표·차트·날짜 선택기 등 PMS에 필요한 부품이 전부 성숙한 라이브러리로 존재. 1인 개발에선 "막혔을 때 답을 찾는 속도"가 생산성의 절반.
2. **인력 풀 최대** — 인수인계·충원 시 React 경험자가 가장 흔하다. AI 코딩 도구 지원도 학습 데이터가 많은 React가 가장 매끄럽다.
3. **단순한 정신 모델** — "화면 = 상태의 함수" 원칙 하나로 일관. TS·Vite와의 조합이 사실상 표준.

**대안 비교.** **Vue** — 학습 곡선은 더 완만하고 훌륭한 대안. 생태계·구인 풀에서만 밀린다(이미 잘 아는 사람이 만들면 Vue도 정답). **Angular** — 풀 프레임워크라 대규모 팀+대형 앱용. 1인 개발 사내 도구엔 과한 장비. **Svelte** — 간결하지만 장기 안정성·인력 검증 부족. **SSR(Thymeleaf 등)** — 조회만 있다면 후보였으나, 가동률 대시보드·배정 편집·챗 위젯·확인 카드처럼 화면 내 상호작용이 핵심이라 SPA가 맞다.

**곁들이는 선택.** **Vite** — 과거 표준이던 CRA(폐기됨) 대비 압도적 개발 서버 속도, 현 커뮤니티 표준. **TypeScript** — 백엔드의 정적 타입 규율을 프론트까지 연장. API 응답 형태를 타입으로 고정해 "백엔드가 필드명 바꿨는데 프론트가 몰랐다"류 사고를 컴파일에서 잡는다. *(재사용한 프로토타입 프론트는 JS로 작성돼 있어, 신규 파일부터 TS로 점진 전환한다 — `docs/conventions/react-ts.md`)*

## 7. 인증 — 세션 vs JWT

**두 방식의 개념.**

- **세션:** 로그인하면 서버가 "출입 기록"을 자기 메모리/저장소에 만들고, 브라우저엔 그 기록 번호(세션 ID)만 쿠키로 준다. 상태가 **서버에** 있다.
- **JWT:** 로그인하면 서버가 사용자 정보를 담아 **서명한 토큰**을 발급하고, 이후 서버는 아무것도 기억하지 않는다. 매 요청의 토큰 서명만 검증한다. 상태가 **토큰에** 있다(무상태, stateless).

**왜 JWT인가.**

1. **무상태 = 운영 단순** — 앱을 재기동해도 로그인이 안 풀리고(서버가 기억하는 게 없으니), 2차에 인스턴스를 늘려도 세션 공유 저장소(Redis 등)가 필요 없다.
2. **MCP 채널과의 통합** — 이 프로젝트 고유의 결정적 이유. AI 호스트가 PMS를 호출할 때 "이 사용자로서"를 증명해야 하는데(토큰 패스스루), JWT는 **자기완결적 신분증**이라 호스트를 거쳐 전달돼도 `/mcp`에서 서명만 검증하면 된다. 세션이라면 세션 저장소를 AI 호스트와 공유하는 이상한 구조가 필요했다. 위임 토큰(audience=pms·단명·channel 클레임)과 PAT까지 같은 검증 지점 하나로 처리된다.

**JWT의 대가(알고 써야 하는 것).** 발급한 토큰은 만료 전까지 **회수가 어렵다**(서버가 기억을 안 하니까). 그래서 만료를 짧게 갖고, 취소가 필요한 PAT에는 `jti`(토큰 ID) + 발급 목록 DB 대조를 추가한다. 또 토큰은 암호화가 아니라 **서명**이다 — 내용은 누구나 읽을 수 있으니 비밀정보를 넣으면 안 된다.

**SSO는?** 1차 범위 밖. 도입 시에도 "본인 확인" 단계만 Entra ID(Teams 계정) 같은 IdP에 위임하고 토큰 발급 체계는 유지하는 방식(OIDC)이라, 인증 경계의 발급부만 교체하면 된다.

## 8. 동시성 — 낙관적 락

**문제.** 두 사람이 같은 레코드를 동시에 수정하면 나중 저장이 앞 저장을 조용히 덮어쓴다(lost update).

**두 전략.**

- **비관적 락:** "충돌 날 거야" — 수정 시작 시 레코드를 잠가 남을 기다리게 함. 확실하지만 대기·잠금 방치(열어두고 퇴근) 문제.
- **낙관적 락 ✅:** "충돌은 드물어" — 잠그지 않고, 저장 순간에 version 번호로 충돌만 감지. 충돌 시 409를 주고 사용자가 최신본을 다시 보고 재시도(reload-and-retry).

**왜 낙관인가.** 40명 중 소수만 갱신하고, 진행률(Project)과 M/M(ProjectAssignment)이 다른 레코드라 충돌 면적 자체가 작다. 드문 충돌을 위해 모두를 기다리게 하는 건 손해. JPA `@Version` 하나로 구현된다. "마지막 저장이 이긴다"를 거부하고 **충돌을 사용자에게 알리고 다시 결정하게 한다**는 것이 핵심 설계 의도. 충돌 잦은 특정 화면만 편집락을 선택 적용할 여지를 남겼다.

**AI 채널에도 동일 적용.** `update_progress` 도구가 version 파라미터를 요구한다 — AI가 조회 후 사용자가 확인하는 사이 남이 고쳤으면 409 → 모델이 에러를 읽고 재조회. 사람이든 AI든 같은 충돌 규칙.

## 9. 알림 전달 — 폴링 vs 푸시(SSE/WebSocket)

**구간을 나눠야 정확하다.** 알림의 **발생**(서버 안)은 이미 이벤트 방식이다(2장). 논점은 **전달**(서버→브라우저)이다. HTTP는 클라이언트가 물어야 답하는 구조라, 서버가 먼저 말을 걸려면 연결을 열어 유지해야 한다(SSE/WebSocket).

**선택: SSE 푸시 (알림 발생 즉시 서버→브라우저 단방향 스트림).** *(개정 2026-07-13 — 당초 주기 폴링이었으나 아래 근거로 변경, PROGRESS 결정 기록 참조)*

| 기준 | SSE ✅ | 주기 폴링 |
|------|--------|-----------|
| 즉시성 | 즉시 | 최대 폴링 주기만큼 지연 |
| 서버 부담 | 사용자별 연결 유지·재연결 관리 — 40명 규모에선 SseEmitter로 감당 가능 (프로토타입 검증 완료) | 더 낮음 (DB 조회만) |
| 챗 UX 시너지 | 챗 응답 스트리밍·진행 상태 표시와 같은 기반 공유 | 별도 구현 |
| 인프라 | Nginx 프록시 버퍼링 해제 설정 필요 | 없음 |
| 스케일아웃 | 다중 인스턴스 시 서버 간 중계(Redis pub/sub) 필요 — 2차 과제 | 어느 서버든 DB만 보면 됨 |

당초 논리는 "30초 지연이 무가치하니 푸시 비용이 남는 장사가 아니다"였다. 변경 근거: ① 프로토타입(2026-07)에서 SseEmitter 기반 푸시가 40명 규모에서 문제없이 동작함을 확인 — 관리 비용이 예상보다 작았다. ② 챗 위젯의 진행 상태 표시("가동률 조회 중...")가 어차피 스트리밍을 요구해, 알림이 같은 기반을 공유하면 추가 비용이 거의 없다. WebSocket이 아닌 SSE인 이유: 알림은 서버→클라이언트 단방향이라 SSE로 충분하고, 재연결·Last-Event-ID가 표준에 내장돼 있다. 발생(적재)과 전달(SSE)은 여전히 분리돼 있으므로, 연결 관리가 부담이 되는 날엔 전달 구간만 폴링으로 되돌릴 수 있다 — 구조는 그대로다.

## 10. 배포 — Docker Compose + Nginx

### 10-1. Docker (Compose)

**개념.** 앱과 실행 환경(런타임·라이브러리·설정)을 컨테이너라는 상자로 포장해, 어디서 실행하든 같은 환경을 보장하는 기술. Compose는 여러 컨테이너(Nginx·앱·DB)를 파일 하나로 정의해 한 번에 올리는 도구.

**왜 쓰나.** ① "내 컴퓨터에선 됐는데" 문제 제거. ② 서버 세팅이 문서가 아니라 코드가 됨 — 1인 개발에서 인프라 지식이 머리가 아니라 파일에 남는다. ③ 한 서버에 여럿을 격리 배치(포트·라이브러리 충돌 없음). ④ 배포=이미지 교체, 롤백=이전 이미지 재기동. ⑤ 로컬 개발도 같은 DB를 compose로 자동 기동(NFR-12). 대안인 "서버에 직접 설치"는 재현 불가능한 수동 절차서를 남기고, Kubernetes는 이 규모에 명백한 과잉이다.

### 10-2. Nginx (리버스 프록시)

**개념.** 외부 요청을 가장 먼저 받아 직접 처리(정적 파일)하거나 뒤(Boot 앱)로 전달하는 관문 웹서버.

**왜 쓰나.** ① TLS 종단 — 인증서 관리를 앱과 분리. ② React 정적 파일 서빙 — 파일 나눠주기는 Nginx가 제일 잘함. ③ `/api` 프록시 — 단일 도메인으로 CORS 문제 원천 차단. ④ **`/mcp` 외부 비노출** — AI 전용 문을 바깥 복도에서 아예 제거(토큰 검증 이전의 방어 한 겹). ⑤ 앱 재배포 중 점검 응답, 2차 다중 인스턴스 시 로드밸런서로 그대로 승격. Boot 앱 단독으로도 기술적으론 가능하지만, 출입 관리를 앱 코드 밖으로 빼는 이 표준 구성이 운영 비용이 더 싸다.

## 11. AI 연동 — MCP와 그 주변 선택들

### 11-1. 왜 MCP인가 (vs 자체 API로 AI에 붙이기)

**개념.** MCP(Model Context Protocol)는 AI(호스트)와 도구(서버)를 잇는 공통 표준 — "AI의 USB-C". 우리가 자체 JSON 규격으로 AI 서버와 PMS를 이어도 동작은 한다. 그럼에도 MCP인 이유: ① 도구 목록·호출·에러의 규격이 표준이라 **어떤 표준 클라이언트(Claude 데스크탑, Inspector 등)로도 즉시 테스트** 가능 — 목업 검증(M-1)이 공짜가 된다. ② AI 호스트를 나중에 교체(자체 → ProLLMOps 등)해도 PMS 쪽은 무수정. ③ 생태계의 외부 도구(Slack·GitHub MCP)를 같은 방식으로 끼울 수 있다.

### 11-2. 역할 배치 — 왜 PMS가 서버이고 AI 호스트가 따로인가

챗 UI가 PMS 화면에 있어도 **도구 선택을 판단하는 지능(LLM)이 있는 곳이 호스트**다. LLM은 별도 AI 서버에 두므로 호스트도 거기고, PMS는 자기 기능을 MCP 서버로 노출하는 제공자다. MCP 서버를 별도 프로세스로 빼지 않고 **기존 Boot 앱에 내장**(`/mcp` 인바운드 어댑터)한 이유: 권한·감사·이벤트·트랜잭션을 재구현 없이 상속하고 배포 단위를 늘리지 않기 위해 — "MCP 어댑터는 REST 컨트롤러의 형제"라는 한 줄이 이 결정의 전부다.

### 11-3. 전송 — 왜 Streamable HTTP인가

MCP 전송은 stdio(같은 기기, 표준입출력)와 HTTP 계열이 있다. AI 호스트와 PMS는 **다른 프로세스/호스트**이므로 stdio는 불가. HTTP 계열 중 구식 HTTP+SSE는 deprecated, 현행 표준이 Streamable HTTP(단일 엔드포인트, 필요 시 스트리밍)라서 선택의 여지가 없는 선택이다.

### 11-4. 인증 — 왜 서비스 계정이 아니라 토큰 패스스루인가

AI 서버에 고정 서비스 계정을 주는 게 편하지만, 그 순간 AI는 모든 데이터를 보는 슈퍼유저가 된다 — 팀원이 챗으로 물으면 남의 본부 데이터가 새는 구조(혼동된 대리인 문제). 그래서 **사용자 토큰이 호스트를 통과해 PMS까지 가고, PMS가 평소처럼 그 사용자로 권한을 판정**한다. "AI가 할 수 있는 일 = 그 사용자가 화면에서 할 수 있는 일"이 구조로 보장되고, MCP용 권한 코드를 한 줄도 새로 짜지 않는다. 7장의 JWT 선택이 이 구조의 전제다.

### 11-5. AI 호스트 구현 — 왜 Spring AI인가

호스트는 Python(LangChain 등)으로도 만들 수 있다. Spring AI(Java)를 권장하는 이유: 백엔드와 같은 언어·빌드·배포 체계(1인 개발의 스택 단일화), ChatClient가 에이전트 루프(도구 호출→결과 반영→재호출)를 내장, MCP 클라이언트 스타터 제공. 사내 ProLLMOps를 호스트로 쓰는 대안은 사용자 토큰 전파가 확인되기 전까지 보류하고, 사규 RAG 같은 **보조 도구**로 붙이는 것부터 시작한다(구현 가이드 부록 A).

## 12. 빌드·테스트 도구

- **Gradle** (vs Maven) — 둘 다 표준. Gradle이 빌드 속도(증분·캐시)와 스크립트 유연성에서 우위, 신규 프로젝트의 다수 선택. Maven이어도 무방한 수준의 취향 차이.
- **JUnit 5 + AssertJ** — Java 테스트의 표준 조합. AssertJ는 가독성 높은 단언문.
- **Testcontainers** — 테스트 때 진짜 PostgreSQL을 Docker로 띄웠다 버림. "테스트는 H2, 운영은 PostgreSQL"의 방언 차이 버그를 제거.
- **Spring Modulith Test** — 모듈 경계 위반을 테스트 실패로 잡는다. "경계는 문서가 아니라 코드로 강제"의 실행 수단(NFR-09).
- **MCP Inspector** — MCP 서버를 GUI로 찔러보는 공식 도구. 목업 검증(M-1)과 도구 개발 중 수동 확인용.
